1. Especificação de Requisitos de Software (SRS)

1.1 Introdução e Visão Geral

O The Muse Studio é um "Personal OS" (Sistema Operacional Pessoal) modular. O objetivo é permitir que o utilizador organize a sua vida através de um dashboard central e subpáginas altamente personalizáveis utilizando uma biblioteca de widgets visuais.

2. Documento de Design Técnico (TDD)

2.1 Arquitetura do Sistema

Para suportar os diversos tipos de visualização (Tabela, Quadro, Galeria), o sistema adotará um padrão de Renderizadores de Widget.

Frontend: React com uma biblioteca de componentes de interface rica (ex: Radix UI ou Headless UI).

Engine de Layout: CSS Grid dinâmico para permitir que widgets de tamanhos diferentes se encaixem.

Estado de Dados: Cada subpágina terá um layout_config que mapeia quais widgets carregar.

2.2 Design de Interface e Componentes

2.2.1 Seletor de Visualização (Widget Picker):

Componente tipo "Grid Modal" que exibe ícones para Tabela, Galeria, Cronograma, etc. Ao selecionar, um novo objeto de configuração é injetado no array da subpágina.

2.2.2 Barra de Navegação Inferior (Bottom Nav):

Ícones fixos (Início, Beauty, Mind, UGC) com estado ativo (highlight) baseado na rota atual.

CSS: position: fixed; bottom: 0; z-index: 1000;.

2.4 Lógica de Notificações e Cronograma

Para o widget de Cronograma e Calendário:

O sistema verifica o timestamp de cada item na data_source.

Utiliza o Web Worker para monitorar horários em segundo plano.

Dispara Service Workers para notificações mesmo com o app fechado (especialmente para Mobile).

2.5 Estratégia de Edição (Inline Editing)

Os widgets de Nota Rápida e Alerta devem usar o atributo contentEditable ou componentes Controlled Input sem bordas, permitindo que o usuário clique e digite imediatamente sem abrir modais secundários.

2. Documento de Design Técnico (TDD)

2.1 Arquitetura e Tech Stack

Frontend: React.js (com Next.js para SEO/Performance).

Base de Dados: Firestore (NoSQL) — ideal para a estrutura flexível de widgets e subpáginas.

Estilização: Tailwind CSS + Framer Motion (para transições suaves entre widgets).

Estado: Zustand ou Redux (para gerir o estado dos widgets ativos e o Dark Mode).

2.2 Estrutura de Componentes

2.2.1 Sistema de Widgets (Widget Engine)

Cada widget é um componente isolado que consome uma data_source.

Exemplo de Widget "Quadro": Mapeia itens com status (To Do, Doing, Done).

Exemplo de Widget "Galeria": Renderiza cartões com imagens e metadados.

2.2.2 Navegação e Sidebar

Sidebar: Componente vertical que colapsa em ícones em ecrãs pequenos.

Bottom Nav: Fixa com z-index: 1000, garantindo acesso rápido aos quatro pilares principais: Início, Beauty, Mind e UGC.

2.4 Lógica do Relógio e Notificações

Relógio: Implementado com um setInterval de 1000ms no componente ClockWidget.

Notificações: Integração com a Web Notifications API. Tarefas no calendário agendadas disparam um Service Worker que verifica eventos pendentes a cada minuto.

2.5 Edição Inline (UX)

Para manter a fluidez, os campos de texto no Alerta e nos títulos das Subpáginas utilizam edição direta (contentEditable), guardando os dados no onBlur para evitar modais de "Salvar".
